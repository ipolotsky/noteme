# Issue #15: Simplify event card, move wishes to separate screen

## What was done
- Removed inline note/wish text previews from the event card (`_build_event_card`). The card now shows only title, date, tags, description.
- Renamed the button from "Related notes" to "Wishes" on the event card.
- Reworked `event_related_notes` handler: now shows all wishes as one continuous text block + "Edit wish" button.
- Added new `event_wish_list` handler (action `wish_list`): shows wishes as inline buttons for selecting a specific wish to edit. Each button navigates to the note view screen via `NoteCb(action="view")`.
- Added i18n keys: `events.wishes`, `events.wishes_empty`, `events.edit_wish`, `events.choose_wish` in both ru.json and en.json.

## Key decisions
- Kept `related_notes` as the internal callback action name to avoid breaking existing deep links or feed references.
- The button on event card uses `events.wishes` i18n key (user-facing rename).
- Empty state handled: if no wishes exist, shows "No wishes yet" text without edit button.
- Back button from wish selection list goes to the wishes text screen, not directly to event card.

## Files changed
- `app/handlers/events.py` - removed text preview from `_build_event_card`, reworked `event_related_notes`, added `event_wish_list`
- `app/keyboards/events.py` - changed button label from `events.related_notes` to `events.wishes`
- `app/i18n/ru.json` - added 4 new keys
- `app/i18n/en.json` - added 4 new keys

## Uncertainties
- The `feed.related_notes` key in notifications/feed was left untouched since it serves a different purpose (digest messages).
- The note view screen (`NoteCb action="view"`) already has edit functionality, so the wish editing flow reuses existing note editing.
